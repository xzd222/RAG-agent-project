# AGENTS.md

本文件是本项目的 Agent 开发规则文档。所有后续代码修改、提示词调整、工具扩展、RAG 改造和 UI 开发都应优先遵循这里的边界与约束。

## 项目定位

本项目是一个基于 LangChain、通义千问模型和 Chroma 向量库的轻量级智能客服 / RAG Agent 原型。

核心目标：

- 提供一个可运行、可扩展、可观测的 Agent 应用骨架。
- 支持基于本地知识库的检索增强问答。
- 支持通过工具调用补充上下文，例如用户信息、日期、天气、业务系统数据等。
- 通过 Streamlit 提供最小可用的交互界面。

非目标：

- 不在当前阶段实现复杂多 Agent 协作框架。
- 不在没有评测集和日志观测的前提下盲目扩展大量工具。
- 不把 demo 逻辑直接升级为生产逻辑，必须先补齐配置、错误处理、测试和边界控制。

## 当前目录边界

- `app.py`：Streamlit UI 入口，只负责用户交互、展示流式输出和调用 Agent 服务。
- `agent/`：Agent 编排层，负责模型、提示词、工具的组装，不直接处理底层文档加载。
- `agent/tools/`：工具定义层，负责把业务能力包装为 LangChain tools。
- `rag/`：RAG 服务层，负责向量库、检索、上下文拼接和基于上下文的回答生成。
- `model/`：模型工厂层，负责 ChatModel 和 EmbeddingModel 的创建。
- `utils/`：通用工具层，负责配置、路径、日志、文件加载、提示词加载等。
- `config/`：配置文件目录，只放非敏感配置。
- `prompts/`：提示词模板目录。
- `data/`：本地知识库原始文件目录。如不存在，可创建。
- `chroma_db/`：Chroma 持久化目录，不应提交到版本库。
- `logs/`：运行日志目录，不应提交到版本库。
- `venv/`：本地虚拟环境，不应提交到版本库。

## 开发原则

1. 先保证链路可运行，再做能力扩展。
2. 配置驱动优先，避免把模型名、路径、检索参数、提示词路径硬编码在业务代码中。
3. Agent 编排层只负责编排，不承载文档解析、向量入库、UI 展示等职责。
4. 工具必须有清晰输入、输出、异常语义，不能把 mock 结果伪装成真实结果。
5. RAG 回答必须基于检索上下文；无法从上下文判断时，应明确说明信息不足。
6. 修改提示词时必须考虑可测试性和稳定性，不写过度宽泛、互相冲突的规则。
7. 中文内容统一使用 UTF-8 编码，避免 GBK/UTF-8 混用导致乱码。

## 编码约束

- Python 代码遵循 PEP 8，命名使用清晰的英文。
- 文件、函数、类名应表达职责，不使用含糊名称。
- 不在 import 阶段执行重型初始化，例如创建向量库、加载大模型、批量读取文档。
- 不在模块全局静默吞异常。必要异常必须写日志，并向上返回明确错误。
- 不使用裸 `except`。
- 不把 API Key、Token、数据库密码写入代码或配置文件。
- 所有路径通过 `utils.path_tool.get_abs_path` 或等价方法统一处理。
- 新增公共函数时，优先补充类型标注。

## 配置约束

- 敏感信息只允许放入 `.env` 或运行环境变量。
- `.env` 不应提交到版本库。
- 模型相关配置归入 `config/agent.yml`。
- RAG 相关配置归入 `config/rag.yml`。
- Chroma 和切片相关配置归入 `config/chroma.yml`。
- Prompt 路径配置归入 `config/prompts.yml`。
- 代码读取的配置 key 必须与 YAML 文件保持一致。
- 新增配置项时，需要提供合理默认值或启动时明确报错。

## 提示词规则

系统提示词应放在 `prompts/main_prompt.txt`。

RAG 总结提示词应放在 `prompts/rag_summarize.txt`，至少包含：

- 用户问题变量。
- 检索上下文变量。
- 基于上下文回答的约束。
- 信息不足时的拒答或澄清策略。
- 引用或来源展示策略。

报告类提示词应放在 `prompts/report_prompt.txt`。

提示词禁止事项：

- 不写互相冲突的角色设定。
- 不要求模型编造事实、来源、时间、用户信息或工具结果。
- 不把工具调用协议写成自然语言幻觉规则，工具调用应由 LangChain tool schema 承担。
- 不把业务密钥、内部路径、真实用户隐私写入 prompt。

## RAG 开发边界

RAG 链路应拆分为四个阶段：

1. 文档加载：读取 txt、pdf 等文件并生成 Document。
2. 文档切片：使用统一的 chunk_size、chunk_overlap 和 separators。
3. 向量入库：写入 Chroma，并记录文件 md5，避免重复入库。
4. 检索问答：检索 top-k 文档，拼接上下文，调用模型生成回答。

约束：

- 文档 md5 必须来自原始文件内容，不允许用 Document 对象代替。
- 检索结果为空时，应返回明确的无上下文响应。
- 上下文拼接必须限制长度，避免超出模型上下文窗口。
- metadata 应保留来源文件、页码或分片信息，方便追溯。
- 向量库路径必须可配置。

## Tool 开发边界

每个工具都必须满足：

- 名称表达能力，不使用过宽泛名称。
- description 用中文清楚说明用途、输入和输出。
- 输入参数尽量结构化。
- 输出为稳定的字符串或结构化对象。
- 外部服务调用必须有超时、错误处理和日志。
- mock 工具必须在 description 或函数名中明确标记。

不允许：

- 工具内部直接读取用户隐私文件。
- 工具无边界访问本地文件系统。
- 工具把异常堆栈直接暴露给最终用户。
- 工具返回与真实世界不一致的随机业务数据，除非明确是 mock。

## Agent 编排规则

- Agent 初始化只负责组装模型、系统提示词和工具。
- 系统提示词必须传入实际字符串，不传函数对象。
- 流式输出应兼容普通消息、工具调用消息和异常消息。
- 中间过程是否展示应由 UI 或配置控制。
- Agent 不直接写入向量库，不直接读取原始文档。

## UI 规则

Streamlit UI 应保持简单、稳定：

- 提供问题输入框。
- 支持流式展示回答。
- 支持显示错误信息。
- 可选显示引用来源。
- 不在 UI 层硬编码模型和检索参数。

UI 不应承担：

- 文档切片逻辑。
- 向量库底层操作。
- 工具实现逻辑。
- 模型初始化细节。

## 日志与错误处理

- 使用 `utils.logger_handler` 统一创建 logger。
- 关键阶段必须记录日志：配置加载、模型初始化、工具调用、RAG 检索、向量入库、异常。
- 用户可见错误应简洁，开发日志应保留详细 traceback。
- 不在正常业务路径中使用 `print` 替代日志。

## 测试要求

新增或修改核心逻辑时，至少覆盖以下最小测试：

- 配置文件能成功加载。
- prompt 文件能成功加载。
- 模型工厂能读取正确配置。
- RAG 服务能完成初始化。
- 文档 md5 去重逻辑正确。
- 工具函数能被直接调用并返回稳定格式。

如果没有测试框架，优先引入 `pytest`，并把测试放入 `tests/`。

## 版本与依赖

- 项目应维护 `requirements.txt` 或 `pyproject.toml`。
- 不提交 `venv/`。
- 不提交 `__pycache__/`。
- 不提交本地日志、向量库和密钥文件。
- 升级 LangChain、Chroma、DashScope 等核心依赖前，需要验证 API 兼容性。

## 安全边界

- 不在代码中泄露 `.env` 内容。
- 不把用户输入拼接为 shell 命令。
- 不允许 Agent 自主执行破坏性文件操作。
- 不允许 Agent 在未确认的情况下访问外部网络服务并产生费用。
- 外部 API 调用必须显式配置凭证来源和超时。

## Git 操作边界

- 任何开发任务开始前，应先给出清晰的执行计划。
- 如果项目尚未初始化 Git，必须先获得用户明确允许，才能执行 `git init`。
- 所有 Git 操作都必须获得用户明确允许后才能执行，包括但不限于 `git status`、`git add`、`git commit`、`git remote`、`git push`、`git pull`、`git branch`、`git checkout`。
- 每完成一个明确步骤后，应在用户允许的前提下创建一次本地提交。
- 每次本地提交完成后，应在用户允许的前提下 push 到 GitHub。
- 不得在未经用户明确要求的情况下执行破坏性 Git 操作，例如 `git reset --hard`、`git clean`、强制 push、删除分支、回滚用户改动。
- 如果远程仓库尚未配置，必须先向用户索取 GitHub 仓库地址，并获得允许后再配置 remote。
- 提交信息应简洁说明本次变更范围，避免把多个无关步骤混入同一次提交。

## 推荐开发顺序

1. 修复编码和明显运行时错误。
2. 补齐配置一致性。
3. 修通 Streamlit -> Agent -> Tool -> RAG 的最小链路。
4. 增加文档入库脚本。
5. 增加最小测试。
6. 增加 README 和 `.gitignore`。
7. 再扩展真实业务工具和多轮记忆。

## 交付标准

一次功能修改完成后，应至少说明：

- 修改了哪些模块。
- 解决了什么问题。
- 如何运行或验证。
- 仍然存在什么限制。

在没有通过基本运行验证前，不应声明功能已经完成。
